# inference.club infrastructure

Pulumi (TypeScript) program that provisions a Hetzner Cloud VPS and deploys
the inference.club app stack (Caddy + Nuxt frontend + Django backend +
Postgres) over `docker compose`. Driven entirely from GitHub Actions — you
don't need Pulumi installed locally.

## Layout

```
infra/
├── index.ts            # entry point
├── server.ts           # Hetzner server + firewall + cloud-init + auto SSH key
├── deployment.ts       # auto-gens app secrets, ships templates over SSH, runs compose
├── config.ts           # typed access to Pulumi config / secrets
└── templates/
    ├── docker-compose.yml.tpl
    ├── Caddyfile.tpl
    └── backend.env.tpl
```

`__TOKEN__` placeholders in templates are substituted at `pulumi up` time
from stack config (no templating library needed).

## Required GitHub repo secrets

Set at <https://github.com/inference-club/inference.club/settings/secrets/actions>:

| Name | Purpose |
|---|---|
| `HCLOUD_TOKEN` | Hetzner Cloud API token (Read & Write). Console → Security → API Tokens |
| `GHCR_TOKEN` | GitHub PAT (classic) with `read:packages` scope. Lets the VPS pull private images from GHCR |
| `PULUMI_ACCESS_TOKEN` | <https://app.pulumi.com/account/tokens> (free tier) |

Optional (only needed when you want GitHub OAuth login working in prod):

| Name | Purpose |
|---|---|
| `GH_OAUTH_CLIENT_ID` | Client ID from a production GitHub OAuth App |
| `GH_OAUTH_CLIENT_SECRET` | Corresponding secret |

Auto-generated by Pulumi (no input needed): SSH keypair, `DJANGO_SECRET_KEY`,
`POSTGRES_PASSWORD`. They live encrypted in Pulumi state and persist across
runs.

## First deploy

1. Make sure the three required secrets above exist in the repo.
2. Push at least one commit to `main` so `build-and-push` produces images
   tagged `:latest` in GHCR.
3. **Actions tab → `infra-deploy` → Run workflow** (manual trigger). Leave
   the `image_sha` blank to use `:latest`.
4. Watch the run; the last log line shows `Outputs: serverIp = "<ipv4>"`.
5. Add two A records at your DNS provider pointing both `inference.club` and
   `api.inference.club` at that IP.
6. Caddy auto-issues TLS certs within ~30 s of DNS resolving. Done.

## Subsequent deploys

- **Automatic** — push to `main`. `build-and-push` builds new images, then
  `infra-deploy` runs `pulumi up` with the new SHA.
- **Manual** — Actions tab → `infra-deploy` → Run workflow. Optional
  `image_sha` input to roll back to a specific build.

## Tearing it down

Actions tab → **`infra-destroy`** → Run workflow → type `destroy` (lowercase)
in the confirmation field. Runs `pulumi destroy` which deletes the Hetzner
VPS and everything on it (Postgres data included). Pulumi state is preserved,
so you can re-run `infra-deploy` from a clean slate.

## Day-2 ops

| Need | How |
|---|---|
| SSH in | The deploy SSH key lives in Pulumi state. From a workflow run log, view the `pulumi up` output, or use `pulumi stack output --show-secrets` locally if you've run `pulumi login`. |
| Tail app logs | SSH in, then `cd /srv/inference-club && docker compose logs -f backend` |
| Force-redeploy current images | Re-run `infra-deploy` from the Actions tab |
| Roll back | Re-run `infra-deploy` with an older `image_sha` input |
| Tear down | `infra-destroy` workflow |

## Deferred — revisit before scaling up

- **Backups.** Postgres data lives in a bind mount on the single VPS. Add
  `pg_dump` cron + offsite copy (Backblaze B2 or similar) before this matters.
- **Monitoring / log shipping.** Currently `docker logs` only.
- **Zero-downtime deploys.** Compose restarts cause ~5 s of downtime; fine
  for MVP.
- **DNS as code.** Adding `@pulumi/cloudflare` would make the DNS records
  part of `pulumi up`. Worth doing once first deploy is proven.
- **Staging environment.** `pulumi stack init staging` + a second set of
  GitHub secrets pointing at a separate stack.
