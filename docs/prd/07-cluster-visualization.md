# PRD 07 — Living Cluster: a 3D visualization of a provider's Kubernetes fleet

**Status: V0 + V1 shipped 2026-06-11 (scene from manifest; agent
`GET /cluster/state` + backend proxy; memory volumes, pod phases, degradation
states). Deferred from V1: request-flow pulses and the per-model request
sparkline — both need a per-model stats source that doesn't exist yet.
V2 (self-generated assets) and V3 (story mode) not started.**

Implementation (cross-repo contract — keep in sync):
- agent: `internal/discovery/clusterstate.go` (`ClusterState` JSON),
  `/cluster/state` route in `main.go`, `discovery: kubernetes` manifest marker
- backend: `ProviderClusterStateView` →
  `GET /api/inference/providers/<id>/cluster/` (30s cache), validator accepts
  optional top-level `discovery`
- frontend: `composables/useClusterState.ts` (`ClusterSnapshot`, poller),
  `components/cluster/{Scene,HostMachine}.vue`, pages
  `/[username]/cluster` (public, commands redacted) + `/dashboard/cluster`
  (owner, commands shown)

## One-liner

A detailed, accurate, real-time Three.js visualization of a provider's
Kubernetes cluster on inference.club — the computers, their memory and GPU
footprints, and the inference apps they run — where the 3D assets themselves
are generated by inference.club's own image and mesh services. The platform
rendering its own substrate, with art it made itself.

## Why

- The kubernetes discovery mode (inference-club-agent `AGENT_DISCOVERY=kubernetes`)
  means the manifest is now *derived truth*: hosts, GPUs, services, exact
  images and commands. That data deserves better than a YAML dump — it can be
  a place you *walk around in*.
- Provider profile pages are inference.club's storefront. "Here is my actual
  cluster, live" is the most honest possible flex — and a reason for providers
  to adopt the k8s agent.
- Dogfooding loop: generating the scene's assets (chassis meshes, GPU cards,
  textures, icons) through /v1/images/generations and /v1/3d/generations makes
  the feature a standing integration test of the platform's own modalities.

## What the user sees

`/[github_login]/cluster` (public, linked from the provider profile) and a
richer authenticated `/dashboard/cluster` for the owner:

- **Machines as machines.** Each node is a 3D computer on a desk/rack floor —
  form factor keyed by node class (tower w/ visible GPU for the 4090 boxes,
  compact gold slab for a DGX Spark, generic box for unknown). Hover/click →
  node card: hostname, IPs, arch, GPU model + VRAM, k8s version, conditions.
- **Apps as cartridges.** Each labeled Service renders as a glowing module
  slotted into its node, colored by modality (reuse the design-system modality
  colors from PRD 05). Click → service card: image, exact command, pod phase,
  restarts, the models it serves, live request sparkline (existing per-model
  stats).
- **Footprints as physical volume.** Memory: each node has a translucent
  volume showing requested/used/allocatable RAM; pods stack inside it like
  cargo. GPU: VRAM bar docked to the GPU card mesh. Live-ish (30–60s poll).
- **Flow.** When requests route through the agent (existing request stream),
  animated pulses travel backend → agent pod → service module. External
  endpoints (LM Studio on a host) render as satellite devices tethered by a
  cable off the cluster floor.
- **Degradation with honesty.** Nodes that are NotReady tip over / power down;
  ImagePullBackOff pods flicker; a wedged service (the TRELLIS incident that
  started all this) smolders. The viz must never show a healthier cluster
  than kubectl does.

## Data path

1. **Shape (already shipped):** the k8s-discovered manifest (hosts, GPUs,
   services, image+command, models). No changes needed for v1 shape.
2. **Live stats (new, small):** agent gains `GET /cluster/state` —
   per-node: allocatable/usage (memory, nvidia.com/gpu), conditions;
   per-pod: phase, restarts, container memory usage. Sources: k3s ships
   metrics-server, so `metrics.k8s.io` is available; node conditions from the
   existing node LIST. Backend proxies it at
   `GET /api/inference/providers/<id>/cluster/` with a short cache, only for
   providers whose latest manifest is k8s-sourced (manifest gains
   `discovery: kubernetes` marker — additive, validator change).
3. **Frontend:** Nuxt page + TresJS or raw Three (NetworkScene.vue and the
   logo/scene components are prior art in-repo; follow whichever pattern
   NetworkScene established). Scene graph driven by a single
   `ClusterSnapshot` type; polling composable `useClusterState`.

## The asset pipeline (the fun part)

Assets are *generated, versioned, and attributed*:

- `design/cluster-assets` page (staff/dev): prompts catalog → generate chassis
  + GPU + accessory meshes via `/v1/3d/generations` (TRELLIS, once the spark
  box is healthy — its rehabilitation doubles as this feature's asset run) and
  textures/sprites via `/v1/images/generations` (FLUX). Each kept asset is an
  ordinary public InferenceRequest — the scene credits its own generation
  requests (click a desk → "generated on club-host, trellis-2, seed 42").
- Fallback primitives: the scene must render fully with procedural boxes when
  generated assets are absent (asset loading is progressive enhancement).
- Static fallbacks live in `frontend/public/design/cluster/` (same pattern as
  the existing `design/jax.obj`).

## Phases

- **V0 (static truth):** scene renders from the stored k8s manifest alone —
  nodes, GPUs, service modules, modality colors, procedural geometry. Public +
  dashboard pages.
- **V1 (live):** `/cluster/state` agent endpoint + backend proxy; memory
  volumes, pod phases, request pulses; degradation states.
- **V2 (self-generated assets):** asset pipeline page, TRELLIS/FLUX assets,
  attribution cards; assets keyed by GPU product / node class.
- **V3 (story mode):** time-scrubbing over manifest history (ServiceManifest
  rows are already versioned) — watch the cluster grow from the first
  lmstudio Service to the full fleet. Pairs with the OpenStory narration of
  the migration.

## Non-goals (v1)

- Not a kubectl replacement: read-only, no actions on the cluster.
- No multi-cluster federation view; one provider = one scene.
- No WebRTC/live streaming of someone's session — that's OpenStory's lane;
  this PRD only renders cluster state.

## Open questions

- TresJS (Vue-idiomatic) vs raw Three like NetworkScene — decide after reading
  NetworkScene's patterns; consistency wins unless it fights us.
- Memory *usage* (not just allocatable) requires metrics-server figures per
  pod; confirm acceptable cardinality in the `/cluster/state` payload caps.
- Does the public page show exact commands? (Owner sees all; public maybe
  image + model names only — commands can leak paths/flags. Default: redact
  args on public view, owner opt-in to show.)
